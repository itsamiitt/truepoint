# Authentication Roadmap (Prioritized)

This document sequences the remaining authentication work into four waves (P0–P3), turning the gaps in [`06-gap-analysis.md`](./06-gap-analysis.md) and [`05-planned-not-built.md`](./05-planned-not-built.md) into ordered, dependency-aware deliverables. It is a synthesis/sequencing document: each item carries a status, a dependency, the owning `truepoint-*` skill, and a single acceptance criterion. **Read this against the code, not against the original snapshot:** the working tree has already advanced past several items the planning docs list as unbuilt — the cross-tenant selector hardening, the `platform_audit_log` lockdown, the `requireOrgRole`/`requireStaffRole` guard family, the `org_role`/`platform_staff` migrations, the workspace-admin session-management API, and the platform-admin staff/impersonation/audit endpoints are now present — so those are scored at their **real** status here, not at the planning docs' "Planned." Status vocabulary is exactly `Implemented | Partial | Stub | Planned | Absent`; every TruePoint claim carries a `file:line` anchor and every external claim a source URL. This roadmap maps onto the phases in [`docs/planning/admin-auth-buildout-plan.md`](../docs/planning/admin-auth-buildout-plan.md).

> **Owning skill, not a named DRI (AUTH-048, DECIDED).** The **Owning skill** column below assigns each item to a `truepoint-*` skill, not a person — keep skill-owner as the assignment model. *Rationale:* it is the repo's deliberate multi-agent operating model (`CLAUDE.md`); when this plan is handed to a delivery team, add a named DRI per item *alongside* the skill — until then the skill-owner column is correct.

> **Two load-bearing facts restated** so a wave is never misread as "almost done": the real OIDC/SAML adapters are **Stub** — they throw and only the mock works off-production (`packages/auth/src/sso/providers.ts:16-47`); the end-user `/account/security` route is **Absent** — `SecurityPanel` only deep-links to a route that does not exist on the auth origin (`apps/web/src/features/settings-user/components/SecurityPanel.tsx:14-26`; there is no `apps/auth/src/app/account/` directory). A wave that depends on either of these is gated on real net-new work, not wiring.

## How the waves are ordered

> **Priority axis — "delivery wave", not "business priority."** The P0–P3 labels in this document are a **delivery wave**: they sequence work by *dependency order* (what must ship before what), not by how badly an enterprise wants it. That is a different axis from the **business priority** (P0–P3) in [`07-priority-matrix.md`](./07-priority-matrix.md), which ranks how badly an enterprise needs each item. The two can disagree: passkeys are **P0 business priority** (enterprises want phishing-resistant MFA) yet land in the **P3 delivery wave** here, because they depend on the P1b `/account/security` enrollment wizard. Read a wave as "when it can ship", and doc 07's column as "how much it's wanted".

- **P0 — Security correctness.** Things that are a liability if left as-is, or that an enterprise security review treats as a blocker regardless of features. Most of the snapshot's "Phase 0" is already done; what survives P0 is a small, finishable set.
- **P1 — Enforcement + user self-service.** Turn the well-resolved policy into an *enforced* login gate, and build the user account-security surface that is currently a dead deep-link. This is where "a stored knob is not an enforced control" gets fixed.
- **P2 — Enterprise federation.** The real OIDC/SAML adapters and SCIM 2.0 + deprovisioning — the two largest enterprise-deal blockers.
- **P3 — Modern / zero-trust.** WebAuthn/passkeys, adaptive/risk-based + step-up, CAEP/Shared-Signals, sign-in alerts, and the remaining platform-admin expansion (the impersonation login-as mint, audit export). Several of these (passkeys above all) are **high business priority in doc 07** but sequence here because they depend on the P1b enrollment wizard — late *wave*, not low *value*.

A guiding rule throughout (from the buildout plan's invariants, `docs/planning/admin-auth-buildout-plan.md:143-150`): **server-side authz on every route, tenant/workspace context only from the session, new tables `tenant_id NOT NULL` + FORCE-RLS, allowlisted write fields, and an audit row in the same transaction.** The roadmap does not relax these; it sequences work that must satisfy them.

## Wave P1 — Enforcement on the login path + user self-service (delivery wave)

> **Delivery wave** — sequenced after P0 because enforcement depends on the (built) resolver and the new self-service route; see the axis note above.

The policy resolver and the admin surface are built; what is missing is the **enforcement** and the **user-facing account-security surface**. This wave fixes the most dangerous illusion in the system — an admin sets a session cap or IP allowlist that silently does nothing. **Lockout-capable enforcement rule:** every control that can lock a tenant out of its own login (session-timeout, allowed-methods, IP-allowlist, `require_sso`) ships behind a **per-tenant default-OFF flag** with a documented **break-glass / disable path** (a platform-staff toggle that re-opens password login), so a misconfiguration is recoverable without a deploy.

## Wave P2 — Enterprise federation (delivery wave; the two biggest deal-blockers)

> **Delivery wave** — federation depends on the (built) interface/token plumbing and unblocks the `require_sso` cutover (P1a); the XL items here are the long poles of the plan.

Both items below are the defining enterprise asymmetry: SSO controls *login*, SCIM deprovisioning controls *removal*. The config surfaces and token plumbing are already wired (`SsoConfigPanel`, `IdentityPanel`, `ssoConfigRepository`, `scimTokenRepository`); what is missing is the protocol substance. **The OIDC adapter, the SAML adapter, and SCIM 2.0 + deprovisioning are XL (multi-week, specialist) items** — they are protocol-correctness work where the security ACs (doc 09) are the ship gate, not afterthoughts.

## Sequencing notes & critical-path callouts

- **P1a is the highest-leverage wave.** The resolver is already correct (`packages/auth/src/policy.ts`); enforcement is mostly wiring it into `flow.ts`/`refresh.ts`. Until it lands, the admin's session-timeout and IP-allowlist controls are a *security illusion* — the gap analysis scores both Critical specifically because the admin reasonably believes a control is real when it is not.
- **P1b (the `/account/security` route) unblocks several P1a/P3 items.** Forced in-login MFA enrollment (P1a), the real MFA catalogue, and passkey registration (P3) all need this route to exist. Build the route shell + TOTP first; it is the spine of user self-service.
- **P2 SSO and P2 SCIM are independent of each other** and can run in parallel, but **deprovisioning depends on SCIM `/Users`** *and* on record-ownership reassignment owned by the data skill — coordinate that handoff early.
- **`require_sso` sequencing is resolved inside the P1a row, not deferred.** Build the gate now behind a per-tenant default-OFF flag; the API rejects *enabling* `require_sso` while `getSsoProvider(protocol)` returns the throwing Stub adapter (`providers.ts:16-47`), so a tenant can only flip it ON after its real adapter passes test-connection (P2).
