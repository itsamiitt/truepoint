# Gap Analysis — Enterprise Benchmark vs. TruePoint

This document scores TruePoint's authentication against the 2025–2026 enterprise benchmark assembled in [`01-enterprise-benchmark.md`](./01-enterprise-benchmark.md) (NIST SP 800-63B-4, OWASP ASVS 5.0, FIDO/CISA, OpenID CAEP/SSF, and the Okta/Entra/WorkOS admin surfaces). The headline is deliberately uncomfortable: at the **library / transaction level** TruePoint's primitives are genuinely strong — Argon2id hashing, EdDSA access tokens with JWKS, rotating refresh tokens with reuse-detection and family revocation, strictest-wins policy resolution, and a clean SSO seam — but the **user-facing self-service surface, the real SSO/SAML/OIDC adapters, SCIM 2.0, and login-path policy enforcement** are the big, enterprise-blocking gaps. Status vocabulary is exactly `Implemented | Partial | Stub | Planned | Absent`; every TruePoint claim carries a `file:line` anchor and every benchmark claim a source URL. Where a primitive exists but is not wired onto the live login path (e.g. policy enforcement — session timeout, IP allowlist) it is scored `Partial`, not `Implemented` — a stored knob that does not gate a real login is not an enforced control.

A note on three load-bearing facts restated from doc 01 so they are never misread: the real OIDC/SAML adapters are **Stub** — they throw (`packages/auth/src/sso/providers.ts:16-38`) and `getSsoProvider` only returns a working (mock) provider off-production (`packages/auth/src/sso/providers.ts:44-47`); the end-user `/account/security` route is **Absent** — `SecurityPanel` only deep-links to a route that does not exist on the auth origin (`apps/web/src/features/settings-user/components/SecurityPanel.tsx:14-26`, and there is no `apps/auth/src/app/account/` directory); SCIM is **token-mint-only** — the SCIM 2.0 endpoints and deprovisioning automation are unbuilt (no `/scim/v2/*` route exists under `apps/`).

---

## How to read the severity column

Severity is the **enterprise-deal impact** of the gap, not the difficulty of closing it:

- **Critical** — blocks an enterprise sale or is an active security/compliance liability (no real SSO, missing real-SSO threat gates, no SCIM deprovisioning, no enforced session timeout, no user MFA self-service).
- **High** — a control an enterprise security review will flag as missing or unenforced (policy enforcement on login, passkeys / phishing-resistant MFA, enforced password policy + breach screening, auth observability/ops, account-recovery hardening). *(Audit immutability is no longer listed here — `platform_audit_log` append-only/RLS is remediated; see section (k).)*
- **Medium** — expected-but-deferrable (adaptive risk, concurrent-session limits, login history).
- **Low** — polish or advanced/aspirational (CAEP/SSF, named locations, conditional-access rule builder).

## Top gaps — ranked

Ranked by enterprise-deal impact (Critical first), with the one-line reason each blocks a sale or fails a security review. This list is re-baselined against the corrections above: `platform_audit_log` (now remediated — section (k)) and the provider-configs / staff-console gap (now Implemented + write-capable — section (i)) are **dropped**, and the real-SSO *threat gates* now lead because they are the gating risk the moment the adapters are wired.

1. **Real-SAML threat gates are Absent.** *Critical.* The instant `@node-saml/node-saml` is wired, the assertion path needs signature/digest validation against pinned IdP metadata, `Conditions`/time/audience/recipient checks, `InResponseTo`/RelayState binding, replay cache, and XML-signature-wrapping defences — none exist yet. A real SSO adapter without these is a worse liability than the current fail-closed Stub. See [`09-threat-model.md` → SAML validation](./09-threat-model.md#saml-validation).
2. **Real SSO adapters are Stub (SAML + OIDC throw in production).** *Critical.* `oidcProvider`/`samlProvider` throw and `getSsoProvider` only returns a working provider off-production (`packages/auth/src/sso/providers.ts:16-47`). No enterprise buys without working SSO against their own IdP. Wire `arctic` (OIDC) + `@node-saml/node-saml` (SAML) behind the existing mock interface — *with* the gates in #1 and the OIDC `id_token`/JWKS/SSRF gates ([`09-threat-model.md` → OIDC id_token validation](./09-threat-model.md#oidc-id_token-validation)).
3. **SCIM 2.0 endpoints + deprovisioning are Absent.** *Critical.* Only token minting exists; there is no `/scim/v2/*` route and no directory-event-driven session revocation / record reassignment. This is the deprovisioning asymmetry doc 01 §(b) calls enterprise-critical — a terminated employee keeps live sessions and owned data. Watch the deprovision-vs-active-session race ([`09-threat-model.md` → SCIM deprovisioning race & token abuse](./09-threat-model.md#scim-deprovisioning-race--token-abuse)).
4. **Auth-policy enforcement on the login path is only partial.** *Critical (in aggregate).* The resolver and the admin UI are built, but **session timeout, IP allowlist, allowed-methods, and require-SSO are stored and resolved, not enforced** (`packages/auth/src/policy.ts:42-55` resolves; the login path does not gate). An admin who sets a session cap or IP allowlist gets a control that silently does nothing — worse than not offering it.
5. **User account-security self-service is Absent.** *Critical.* `SecurityPanel` deep-links to a `/account/security` route that does not exist (`apps/web/.../SecurityPanel.tsx:14-26`) and fakes the MFA catalogue (`enrolled: false`). No self-serve password change, MFA enroll/disable, session list, or login history — table stakes for B2B SaaS, and the P1b wizard that several later gaps (passkeys) depend on.
6. **Auth observability & operations are Absent.** *High.* No structured auth-event metrics/SLIs, no SLO/error-budget for the login path, no signing-key-rotation / key-compromise / KMS-custody runbook, no auth-specific incident-response & breach-notification path. An enterprise expects auth to be *operable*, not just functional. See [`10-operations-and-compliance.md` → Auth observability & SLIs](./10-operations-and-compliance.md#auth-observability--slis) and [→ Incident response & breach notification](./10-operations-and-compliance.md#incident-response--breach-notification).
7. **DSAR / retention / deletion of auth artifacts is unplanned.** *High.* Auth events, sessions, and MFA secrets are not enrolled in any erasure/retention path, and the immutable-audit-vs-erasure tension is unresolved. See [`10-operations-and-compliance.md` → DSAR, retention & deletion of auth artifacts](./10-operations-and-compliance.md#dsar-retention--deletion-of-auth-artifacts).
8. **No phishing-resistant MFA (passkeys/WebAuthn).** *High.* `verifyMfaCode` routes `webauthn` to the non-TOTP fall-through that returns `false` (`packages/auth/src/mfaVerify.ts:22`). TOTP MFA already exists, so this is High (lacking a phishing-resistant *option*), not Critical; CISA names FIDO the strongest tier. Sequenced **P3 (delivery wave)** because passkey enrollment depends on the P1b `/account/security` wizard (#5).
9. **MFA `required` blocks but cannot enroll mid-login.** *High.* `finalizeLogin` throws `mfa_required` (`packages/auth/src/flow.ts:152-160`) with no forced in-login enrollment screen — a required-MFA org with un-enrolled members locks them out.
10. **No per-user self-service session/device visibility; workspace Members API still UI-only.** *High.* The workspace-**admin** session surface is *built* — owner/admin can list members' sessions, revoke one, and force-reauth (`apps/api/src/features/workspaces/sessionRoutes.ts`, mounted `app.ts:71`; core `adminSessions.ts`). What is still missing: (i) the **per-user** `/account/security` route where a user views/revokes their *own* sessions (the route does not exist), and (ii) the workspace **Members** API — the members admin UI hits `/workspaces/current/members`, which returns 404, so it is UI-only (`apps/web/src/features/settings-workspace/api.ts:43-80`).
11. **No breached-password screening or enforced password policy.** *High.* The Argon2id hash is right (`password.ts:5-26`) but the enforced min-length and breach-screening on set are missing — NIST [SP 800-63B-4 §3.1.1.2](https://pages.nist.gov/800-63-4/sp800-63b.html) makes both a **SHALL** ([ASVS V6.2](https://github.com/OWASP/ASVS/blob/master/5.0/en/0x15-V6-Authentication.md) echoes it). Sequenced **P0 (delivery wave)**.
12. **No adaptive/risk-based auth or step-up.** *Medium.* Signals are captured (`session.ts:26-28`) but unscored; no new-device challenge, impossible-travel, or re-auth-before-sensitive-action.
13. **No CAEP / Shared Signals.** *Low.* Internal revocation is fast (`session.ts:71-81`) but there is no cross-service signal exchange ([OpenID CAEP](https://openid.net/specs/openid-caep-1_0-final.html)); most B2B SaaS also lack it.

> **Closed since the last baseline (kept here so the delta is legible, not as open work).** `platform_audit_log` tamper-resistance is **remediated** — schema + deny-all RLS + append-only trigger + `leadwolf_app` REVOKE all ship (`packages/db/src/rls/platform.sql:16-49`, `packages/db/src/applyMigrations.ts:78`); keep the regression test green. The platform/staff **provider-configs API and staff RBAC** are **Implemented and write-capable** (`apps/api/src/features/admin/providerConfigs.ts`, mounted `apps/api/src/features/admin/routes.ts:248`, gated `requireStaffRole('super_admin')`); the only residual governance gaps are the impersonation login-as token mint (Partial) and the audit-log export (Absent).

**Bottom line.** TruePoint's authentication *core* is unusually solid for its stage — the token/session/hashing/policy-resolution primitives are correct, tested, and senior-quality. The gap is almost entirely at the **edges that an enterprise actually touches**: real SSO, SCIM deprovisioning, user self-service, and turning the well-resolved policy into a *enforced* gate on the login path. The build-out is therefore less "invent the hard parts" and more "wire the strong primitives to the real surfaces and stop trusting a stored knob to be an enforced control."
