# 0.6b ‚Äî Cross-origin MFA-status read (design)

**AUTH-068 ¬∑ status: design done, implementation held for security review ¬∑ 2026-07-07**

Goal: apps/web's `SecurityPanel` (app.truepoint.in) shows the user's **true** two-step On/Off state instead of
deferring entirely to the auth origin (0.6a removed the *fake* `enrolled:false` badges; this restores real state
without lying). NOT urgent ‚Äî 0.6a already killed the misinformation; this is an enhancement.

## The hard constraint found by read-first (why the obvious approach is wrong)

`user_mfa_methods` (packages/db/src/schema/auth.ts:222) has **no `tenant_id`** ‚Äî it is keyed by `user_id` and is
**auth-service-owned**. The RLS model is explicit (packages/db/src/rls/auth.sql:73‚Äì78):

> The customer app's non-BYPASSRLS `leadwolf_app` role only ever sees rows for the active tenant via the GUC. The
> user-scoped auth tables (`user_sessions`/`user_mfa_methods`/`trusted_devices`/`auth_email_tokens`) are ‚Ä¶
> auth-service-owned, keyed by user_id.

So **apps/api (which connects as `leadwolf_app`, tenant-GUC-scoped) cannot read `user_mfa_methods`** ‚Äî the table
has no tenant GUC, so the app role sees no rows. Putting the read on apps/api would require *granting the general
tenant API access to auth-owned auth-factor data* ‚Äî a deliberate separation-of-duties widening, and a security
regression. ‚ùå Rejected. (Implementation must still verify the exact `leadwolf_app` grants before relying on this.)

## Recommended architecture: a read on the AUTH origin, consumed by apps/web via the access token

The auth origin (apps/auth) already owns this data under its privileged connection and already authenticates the
user. Add a small **read** there and have apps/web fetch it:

- **Endpoint:** `GET /auth/account/mfa-status` (auth origin, under the `/auth` basePath).
- **AuthN:** the **access token** (`Authorization: Bearer ‚Ä¶`). apps/web already holds it (it Bearer-calls apps/api).
  The auth origin verifies it (it is the issuer; audience ‚àà `appOrigins()`). *Not* the session cookie ‚Äî it is
  HttpOnly + SameSite and will not be sent cross-site from app.* ‚Üí auth.*.
- **Scope:** strictly `claims.sub` (the token subject) ‚Äî a user reads only their OWN status. Never an id from the
  request. Because MFA rows are user-global (not tenant-scoped) and the read is self-scoped, there is **no
  cross-tenant leak** by construction.
- **Payload:** booleans/counts ONLY ‚Äî e.g. `{ hasVerifiedFactor: boolean, hasTotp: boolean, recoveryCodesRemaining: number }`.
  NEVER secrets, method ids, labels, or timestamps (those stay on the auth-origin surface). Reuses
  `userRepository.listMfaMethods` + `countRecoveryCodes` (already user-scoped).
- **CORS:** credentialed allow-list = the app origins only (mirror the existing `/token/exchange` CORS).
- **apps/web:** `SecurityPanel` fetches it client-side (it is already `"use client"`), renders true "On/Off" and the
  recovery-codes remaining, and keeps the "Manage on the sign-in site" deep-link. Degrade to the current honest
  copy (0.6a) on any failure/offline ‚Äî never block the panel.

## Why this needs the human security review the plan flags

It creates a **new cross-origin data-exposure surface**: an auth-origin endpoint that accepts **app-audience**
Bearer tokens and reveals **auth-factor presence**. Points a reviewer must sign off:
1. Is the auth origin accepting a token minted for the *app* audience acceptable here (issuer=self, aud=app)? (It is
   the same token apps/api trusts; the auth origin is merely another verifier. Confirm the audience check.)
2. Is exposing "does this user have MFA" to the app origin acceptable? (It is the user's own status, self-scoped ‚Äî
   low sensitivity ‚Äî but it is still auth-factor metadata leaving the auth boundary. Booleans-only bounds it.)
3. Rate-limit / caching posture (a cheap read; add a short client cache to avoid a per-render round-trip).

## Safe implementation plan (when greenlit)
1. `userRepository.mfaStatus(userId)` ‚Äî a summary read (thin wrapper over the existing list + count), unit/itested.
2. Auth-origin route `GET /auth/account/mfa-status` ‚Äî token-verified, self-scoped, booleans-only, CORS to app origins.
3. apps/web `SecurityPanel` ‚Äî fetch + render true state; fall back to the 0.6a honest copy on failure.

**Recommendation:** implement under review (or fold into the Phase-1 centralized-IdP work, which formalizes the
auth‚Üîapp read surfaces anyway). Do NOT ship it unsupervised from the autonomous loop ‚Äî it is a new auth-boundary
data exposure, and 0.6a already removed the only actual *bug* (the lie).
