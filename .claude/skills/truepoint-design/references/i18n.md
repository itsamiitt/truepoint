# Internationalization and Localization

TruePoint serves users across many countries and locales. "Millions of users"
globally means the UI must translate, format per locale, and survive both longer
strings and right-to-left scripts. Localization is built in from the start —
retrofitting it means touching every string, every date, and every layout, which is
far more expensive than doing it as you go. This is the design-side discipline; data
residency and which *region* a tenant's data lives in is a separate concern in
**truepoint-security** (compliance).

> **Implementation status:** there is **no i18n layer yet** — no message catalog, no
> i18n library in any package, and today's copy is English literals (including
> inside `@leadwolf/ui`). The enforceable rule today is **translation-readiness**:
> interpolation-shaped strings, no concatenated sentences, no length-dependent
> layout. The catalog, plural handling, and locale formatting below are the target
> to build toward — do not invent an i18n stack mid-feature.

---

## All User-Facing Copy Is Translatable

- **No hardcoded user-facing strings in JSX.** Every label, button, message, empty
  state, and error goes through the translation layer (a message catalog keyed by an
  ID), not as a literal string. A literal string in the UI is untranslatable and is
  the anti-pattern to catch (it pairs with the writing.md "no filler" rule — both are
  about copy being real *and* localizable).
- **Keys with context, not English-as-key** where it matters — so translators know
  what a string is for, and so the same English word that needs two different
  translations can have two keys.
- **Interpolation, not concatenation.** "Added {count} prospects" is one translatable
  unit with a placeholder; "Added " + count + " prospects" cannot be translated
  (word order differs across languages). Never build a sentence by concatenating
  translated fragments.
- **Pluralization is locale-aware.** "1 prospect" vs "2 prospects" — and languages
  with more than two plural forms — go through the i18n library's plural handling,
  not an `if (count === 1)` ternary.

---

## Layout Survives Translation

Translated strings are often longer (German, Finnish) or shorter than English, and
the layout must not break:

- **No fixed-width containers sized to the English string.** Buttons, labels, and
  nav items flex to their content; text truncates with ellipsis + a tooltip where
  space is genuinely constrained (and the full text stays accessible).
- **Don't assume string length** for layout math. A label that fits on one line in
  English may wrap in another language — design for wrapping, don't clip.
- Test key surfaces with a long-string pseudo-locale to catch overflow before it
  ships.

---

## Right-to-Left (RTL)

Languages like Arabic and Hebrew render right-to-left, which mirrors the layout:

- **Use logical CSS properties** (inline-start/inline-end, margin-inline) rather than
  hardcoded left/right, so the layout mirrors automatically in RTL. The sidebar,
  drawers (which `side` they open from), icons with direction (chevrons, arrows), and
  text alignment all flip.
- **Directional icons mirror**; symmetric ones don't. A "back" chevron points the
  other way in RTL; a settings gear doesn't change.
- The design tokens and spacing scale are direction-agnostic; the *application* of
  them uses logical properties so one layout serves both directions.

---

## Locale-Aware Formatting

Numbers, dates, currencies, and names are formatted per the user's locale, never
hardcoded to one convention:

- **Numbers**: separators differ (`1,000.5` vs `1.000,5` vs `1 000,5`). Use the
  platform's locale-aware number formatting. (Tabular figures still apply for
  alignment — see tokens.md.)
- **Dates and times**: format and order differ (MM/DD vs DD/MM vs ISO), as do month
  and day names. Format by locale and the user's timezone — never assume the
  server's. "Saved 3 minutes ago" (relative time, writing.md) is itself localized.
- **Currency**: symbol, placement, and decimal conventions are locale/currency
  specific — deal values are formatted accordingly.
- **Names and addresses**: name order and address format vary by locale — don't
  hardcode "First Last" or a US address shape.

---

## Where It Connects

- **Copy** (writing.md) is written to be localizable: plain, specific, no idioms that
  don't translate, no concatenation.
- **Performance** (large-data.md): translation catalogs are loaded per locale, not
  all locales at once — the user downloads their locale.
- **Data**: which language a *prospect's* data is in is content, separate from the
  *UI* locale; and which *region* a tenant's data resides in is residency
  (**truepoint-security** compliance), separate from localization.

---

## Checklist

- Does every user-facing string go through the translation layer (no hardcoded JSX
  literals)?
- Are sentences built with interpolation + locale-aware pluralization, never
  concatenated fragments?
- Does the layout flex for longer/shorter translations without clipping?
- Does the UI use logical CSS properties so it mirrors correctly in RTL, with
  directional icons flipping?
- Are numbers, dates/times (per user timezone), currencies, and names formatted by
  locale rather than hardcoded?
- Is the user's locale catalog loaded on its own, not all locales at once?
