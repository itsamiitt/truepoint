# design-sync notes — @leadwolf/ui → "TruePoint Design System"

Repo-specific gotchas for syncing `packages/ui` (`@leadwolf/ui`) to claude.ai/design. Read this before any re-sync.

## Shape & why the build is custom
- **Package shape, NO Storybook, NO dist.** `@leadwolf/ui` exports raw TS source (`exports["."]: "./src/index.ts"`) with no build script. The converter's default path here yields **zero components + empty prop contracts** (it reads `.d.ts`/the export entry, of which there are none, and `resolveDistEntry` resolves the `.ts` so synth-mode's source-scan never fires).
- Fix = a reproducible scratch build, `node .design-sync/build-ui.mjs`, recorded as `cfg.buildCmd`. It produces gitignored `packages/ui/dist/` (a fake "built package") + node_modules placements. **The repo's real source/config is never modified.**

## What build-ui.mjs does (re-run before the converter every sync)
1. `tsc` full emit (JS + `.d.ts`) → `packages/ui/dist/`. Needs `allowImportingTsExtensions` + `rewriteRelativeImportExtensions` (TS 5.7+) because source imports carry `.ts`/`.tsx` extensions. `noEmitOnError:false` so it emits even if strict type errors surface.
2. Writes `dist/package.json` `{name, module, types}` so the converter resolves the entry/types.
3. Ships **Geist** (variable woff2 from `apps/auth/node_modules/geist`) + `_fonts.css`. Geist is normally injected at runtime by `next/font` in the apps; the DS bundle has no app, so we ship it for in-brand previews.
4. **Tailwind v4 compile** of `tokens.css` + `theme.css` + `primitives.css` → `dist/_compiled.css` (= `cfg.cssEntry`). Required because the shadcn primitives (`src/components/ui/*`) use Tailwind utility classes generated by the app's `@import "tailwindcss"` — they'd render unstyled otherwise. Input lives in `.ds-sync/_tw-input.css` so `@import "tailwindcss"` resolves from the staged CLI.
5. Copies `react-dom` + `scheduler` (both 19.0.0) into `packages/ui/node_modules` — `vendorReact` needs them under `--node-modules` (React 19 has no UMD → it esbuild-bundles react/react-dom/scheduler).

## Converter invocation (from repo root)
```
node .ds-sync/package-build.mjs --config .design-sync/config.json \
  --node-modules packages/ui/node_modules --entry packages/ui/dist/index.js --out ./ds-bundle
node .ds-sync/package-validate.mjs ./ds-bundle
```
- `--node-modules` is **packages/ui/node_modules** (has react + the DS deps + react-dom/scheduler after build-ui). NOT repo root (sparse — bun isolates) and NOT apps/auth (lacks the DS deps).
- `cfg.srcDir: "../src"` points enrichment (JSDoc → prompt.md, grouping) at the REAL source while contracts come from `dist`'s `.d.ts`.

## Staging deps (.ds-sync)
```
(cd .ds-sync && npm i esbuild ts-morph @types/react @tailwindcss/cli@4.3.1)
```
`@tailwindcss/cli@4.3.1` matches the repo's tailwindcss (apps/auth). Re-stage `.ds-sync` (cp the skill scripts) + reinstall on a fresh clone.

## Components & grouping
- ~43 exports. Two families: token-driven (`.tp-ui-*` classes in primitives.css + inline styles) and shadcn primitives (`ui/*`, Tailwind). `ToastProvider` is a context provider; `RadioOption` only renders inside `RadioGroup`; `Toast`/`useToast` need `<ToastProvider>` — author those previews composed within their parent/provider.
- **Only Toast uses React context** — so NO global `cfg.provider`. Everything else renders standalone.
- Grouping is flat (`general`) for now: dir-based grouping collapses (`components`/`ui` are generic dirs) and there are no doc frontmatter categories. Acceptable for v1; could be enriched later via per-component doc stubs with `category` frontmatter (cfg.docsMap).

## Preview authoring learnings (apply when authoring/regrading)
- **Imports**: `import { X } from "@leadwolf/ui"` (resolves to the `window.TruePointUI` bundle global). `jsx: automatic` — no `import React` needed. Each named PascalCase export = one graded cell. Realistic content only.
- **Overlays with `position:fixed`** (Dialog, Drawer; Toast's toaster) collapse/clip in a card because single-mode's transform wrapper has zero height. Wrap them in a sized stage: `<div style={{position:"relative", height:460, transform:"translateZ(0)", overflow:"hidden", borderRadius:8, background:"var(--tp-surface)"}}>…</div>`, and set `cfg.overrides.<Name> = {cardMode:"single", viewport:"720x520"}`. (Dialog ✓ verified this way.)
- **Floating** (Popover, DropdownMenu, Tooltip) use `position:absolute` inside `.tp-ui-anchor` — render them via their trigger/anchor with surrounding space; the open state is what to show. Likely `cardMode:"single"` too.
- **Wide** components (DataTable, full-width bars) → `cfg.overrides.<Name> = {cardMode:"column"}` (one story per row, full width). DataTable ✓ verified.
- **Context**: only Toast needs `<ToastProvider>` — compose Toast/useToast previews inside it. `RadioOption` renders inside `RadioGroup`.
- Solo-calibrated good: TpButton, Button (Tailwind ✓), DataTable, StatTile, Dialog. Tailwind utilities + Geist font + tokens all render correctly in authored previews.

## Component API facts (from authoring — the emitted .d.ts is LOSSY; trust src/*.tsx)
The emitted `<Name>.d.ts` filters DOM/React-inherited props, so native-element wrappers show only own props (e.g. TpInput → just `invalid`). The runtime accepts all native attrs; the prompt.md examples demonstrate real usage. Author against `packages/ui/src/components/**`, not the d.ts. Key non-obvious APIs:
- **Tp* controls** (controls.tsx): native-attr wrappers. Use `defaultValue`/`defaultChecked` in previews (avoid controlled-without-onChange warnings). TpSelect placeholder = `defaultValue="" ` + leading `<option value="" disabled>`. TpCheckbox `indeterminate` needs a ref (not static).
- **RadioGroup/RadioOption** (ui/radio-group.tsx): NATIVE radios, not Radix. RadioGroup = `<div role=radiogroup>` + children (no value/onChange). RadioOption = native `<input type=radio>` taking `name`/`value`/`defaultChecked`/`disabled` + `children` for row text. Same `name` per group; one `defaultChecked`. Selection highlight is CSS `:has(:checked)`.
- **Pagination** (Pagination.tsx): CURSOR pager — `hasPrev`/`hasNext`/`onPrev`/`onNext`/`label` (label carries the range text). NOT numbered.
- **Icon** (Icon.tsx): takes a lucide component, `icon={Mail}` (import from `lucide-react`), + `size`/`strokeWidth`; color inherits.
- **Avatar**: initials-only (derives from name/email), monochrome, no image src.
- **StatusBadge** tones: success / warning / danger / muted (default muted). No info/neutral.
- **Spinner** ring color is hardcoded (won't inherit color) — author on light surfaces only.
- **Progress** at value 0 renders empty — use non-zero values.
- **Badge** `success` variant = subtle cobalt-50 tint (low contrast but legible).

## Preview force-open patterns (overlays/floating with no controlled `open` prop)
- **Drawer / Dialog / ToastProvider** (`position:fixed`): wrap in a sized `transform:translateZ(0)` stage (height 400–480, overflow hidden) so the fixed layer anchors to the card; render open (`open`, `onClose={()=>{}}`). ToastProvider: an inner component calls `useToast()` and fires toasts with `duration:0` in a mount effect (the `success`/`error` helpers auto-dismiss; use `toast({tone,…,duration:0})`).
- **Popover / DropdownMenu**: open is internal; the trigger render-prop exposes `toggle`. Force-open by stashing `toggle` in a ref and calling it once in a mount `useEffect`. Pad the stage ~200px below for the downward panel.
- **Combobox**: force-open by `.click()`-ing the trigger button via a ref on mount.
- **Tooltip**: opens on hover/focus. Force-open by FOCUSING the trigger button on mount (`ref.current?.querySelector("button")?.focus()`) — React's onFocus = bubbling `focusin`, which a programmatic `.focus()` fires. A dispatched native `mouseenter` does NOT work (React derives onMouseEnter from mouseover). Tooltip opens upward → pad the stage top ~150px. Trigger MUST be focusable (a button, e.g. TpIconButton — not a bare span).
- Card-mode overrides for these live in config.json `overrides` (Dialog/Drawer/Popover/DropdownMenu/Tooltip/ToastProvider = single + viewport; DataTable = column). After ANY change to those overrides, run a full `package-build` before scoped preview-rebuild (else `[CONFIG_STALE]`).

## Build/upload gotchas learned during the first sync (IMPORTANT for re-sync)
- **Previews must NOT import from `lucide-react`.** esbuild parses lucide's 1000+-icon barrel per importing preview, which HANGS the preview compile under memory pressure (it only succeeds when RAM is plentiful). Use the local inline-SVG glyphs in `.design-sync/previews/_glyphs.tsx` instead (they match the `Icon` IconComponent shape, so `<Icon icon={Users}/>` still demonstrates the real API). EmptyState/Icon/StateSwitch use these. If you add a preview that needs an icon, add a glyph to `_glyphs.tsx` — never `import from "lucide-react"`.
- **Memory pressure can hang the full `package-build`** at the 43-preview compile stage (this box is 8GB and often has ~2GB free with other projects' vitest/tsc running). Robust workaround when it stalls: (1) `mv .design-sync/previews .design-sync/previews.hold`, (2) full `package-build` (compiles 0 previews → fast, low memory; produces bundle + contracts + README-with-header + anchor), (3) move previews back (watch out: package-build recreates an empty `.design-sync/previews/`, so `mv previews.hold/*.tsx previews/` then rmdir), (4) `preview-rebuild` ALL 43 in fresh-process batches of ~6-10 (each process reclaims esbuild memory on exit). Then `package-validate`. Each `preview-rebuild` invocation patches the anchor for its components, so rebuild every component (a killed mid-batch run leaves those components' anchor renderHashes stale → `[SYNC_STALE]`; just re-run preview-rebuild for them).
- **`_glyphs.tsx`** lives in `previews/` and is NOT a component, so `package-build` logs a harmless "stale preview: _glyphs" line — ignore it.
- **GRID_OVERFLOW**: the wide/panel/form components are pinned to `cardMode:"column"` and the overlays to `cardMode:"single"` in config `overrides` (10 column + 6 single + DataTable column). cardMode is presentation-only (not in the grade key), so changing it doesn't re-grade; but it DOES change the html → re-run `preview-rebuild` for those after editing, and a full build is needed after adding/removing a `viewport` (viewport IS keyed → `[CONFIG_STALE]`).
- **Transient design-API 404 / empty `list_projects`** happened once mid-upload (design-auth lapse — watch for repeated "Upgraded your claude.ai login" notices). The project + planId were fine after a re-`list_projects`; just retry. Don't recreate the project on a single 404.

## Re-sync risks (what can silently go stale)
- **`dist` is regenerated, not committed** — a re-sync MUST run `cfg.buildCmd` first or the converter sees a missing entry.
- **react-dom/scheduler copies** in `packages/ui/node_modules` are removed by `bun install --force`/clean. build-ui.mjs is idempotent and re-copies if missing.
- **Geist version pinned via apps/auth/node_modules/geist** (1.3.1). If that path moves, font copy is skipped (warns) → previews fall back to system fonts.
- **Tailwind compile pulls `@tailwindcss/cli` from `.ds-sync`** — version drift from the repo's tailwindcss (4.3.1) could change generated utilities; keep the pin in sync.
- **`rewriteRelativeImportExtensions`** is TS-5.7+ only. If the repo downgrades TS below 5.7, build-ui's tsc emit breaks (fall back to emitDeclarationOnly + a re-export index.js wrapper).
